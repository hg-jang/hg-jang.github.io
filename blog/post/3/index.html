    HTTP - 1

    2023-02-13
    • http
    • 공부

    수많은 이미지, 동영상, 문서 등이 하루에도 수십억 개씩 인터넷을 돌아다닌다. HTTP는 이들을 나르기 위한 현대 인터넷의 공용어이다.

    웹 클라이언트와 서버

    앞서 말한 이미지, 동영상 등 웹 콘텐츠는 웹 서버에 존재한다. 웹 서버는 HTTP 프로토콜로 의사소통하기 대문에 HTTP 서버이기도 하다. 웹 서버는 데이터를 저장하고, HTTP 클라이언트가 요청하는 데이터를 제공한다.

    클라이언트가 서버에게 HTTP 요청을 보내고 서버는 요청된 데이터를 HTTP 응답으로 돌려준다.

    이것은 웹의 기본 요소이다.

             → HTTP 요청: index.html 문서를 주세요. →
Client                                          Server
         ← HTTP 응답: index.html 여기 있습니다. ←

    앞서 말한 클라이언트는 우리가 매일 사용하고 있는 크롬, 엣지, 혹은 과거의 인터넷 익스프롤러 같은 브라우저를 말한다. 브라우저는 보이지 않는 서버에게 HTTP 요청을 보내주고 응답 받은 내용을 사용자의 화면에 보여준다.

    리소스

    웹 서버는 웹 리소스를 관리하고 제공한다. 가장 단순한 리소스는 정적(static) 파일이다. 정적 파일은 텍스트 파일, HTML 파일, 워드, JPEG 이미지 등을 포함한다. 그러나 리소스가 반드시 정적일 필요는 없다. 리소스는 요청에 따라 콘텐츠를 생산하는 프로그램이 될 수도 있다. 이들 동적 컨텐츠는 사용자가 누구인지, 언제, 어떤 정보를 요청했는지에 따라 다른 콘텐츠를 생성한다.

    미디어 타입

    인터넷은 수천 가지 데이터 타입을 다루기 때문에, HTTP는 웹에서 전송되는 객체 각각에 MIME(Multipurpose Internet Mail Extensions, 다목적 인터넷 메일 확장) 타입이라는 데이터 포맷 라벨을 붙인다. 웹 서버는 모든 HTTP 객체 데이터에 MIME 타입을 붙인다. 브라우저는 서버로부터 객체를 돌려받을 때 MIME 타입을 통해 다룰 수 있는 객체인지 확인한다.

    Client   ← [ Content-type: image/jpeg ] ←   Server
// MIME 타입: image/jpeg

    MIME 타입은 '/'로 구분된 주 타입과 부 타입으로 이루어진다.

    • HTML로 텍스트 문서: text/html
    • plain ASCII 텍스트 문서: text/plain
    • JPEG 이미지: image/jpeg
    • GIF 이미지: image/gif
    • 애플 퀵타임 동영상: video/quicktime
    • 마이크로소프트 파워포인트 프레젠테이션: application/vnd.ms-powerpoint

    URI

    웹 서버 리소스는 각자 이름을 갖고 있기 때문에, 클라이언트는 관심 있는 리소스를 지목할 수 있어야 한다. 리소스 이름은 URI(Uniform Resource Identifier, 통합 자원 식별자)라고 불린다. URI는 인터넷의 우편물 주소로, 리소스를 고유하게 식별하고 위치를 지정할 수 있다. HTTP는 주어진 URI로 객체를 찾아온다.

                 http://www.anon.com/users/johnDoe/profile.jpeg
client   ← [ Content-type: image/jpeg; Content-length: 8572 ] ←   www.anon.com
// http: http 프로토콜을 사용할 것
// www.anon.com: www.anon.com으로 이동할 것
// /users/johnDoe/profile.jpeg: /users/johnDoe/profile.jpeg라고 불리는 리소스를 가져올 것

    URI에는 두 가지가 있는데, URL과 URN이다.

    URL

    통합 자원 지시자(Uniform Resource Locator, URL)는 리소스 식별자의 가장 흔항 형태이다. URL은 특정 서버의 한 리소스에 대한 구체적인 위치를 서술하는데, 정확히 어디에 있고 어떻게 접근할 수 있는지를 알려준다. 앞선 예시에서 보듯이 대부분의 URL은 세 부분으로 이루어진 표준 포맷을 따른다.

    • URL의 첫 번째 부분은 스킴(scheme)이라고 불리는데, 리소스에 접근하기 위해 사용되는 프로토콜을 서술한다. ex. http://
    • 두 번째 부분은 서버의 인터넷 주소를 제공한다. ex. www.anon.com
    • 마지막을 리소스를 가리킨다. ex. /users/johnDoe/profile.jpeg

    URN

    URN(Uniform Resource Name, 유니폼 리소스 이름)은 콘텐츠를 이루는 한 리소스에 대해, 그 리소스의 위치에 영향 받지 않는 유일무이한 이름 역할을 한다. URN은 위치 독립적이다. 리소스가 그 이름을 바꾸지 않는 한, 여러 종류의 네트워크 접속 프로토콜로 접속해도 문제없다.

    트랜잭션

    HTTP 트랜잭션은 요청 명령과 응답 결과로 구성되어 있다.

    // HTTP 요청 메시지는 명령과 URI를 포함
Client
  → GET /users/johnDoe/profile.jpeg
    Host: www.anon.com


                                      www.anon.com
                 HTTP/1.0  200  OK ←
          Content-type: image/jpeg
              Content-length: 8572
// HTTP 응답 메시지는 트랜잭션의 결과를 포함

    메서드

    HTTP는 HTTP 메서드라고 불리는 여러 가지 종류의 요청 명령일 지원한다. 각각의 HTTP 요청 메시지는 한 개의 메서드를 갖는다. 메서드는 서버에게 어떤 동작이 취해져야 하는지 말해준다.

    • GET: 서버에서 클라이언트로 지정한 리소스를 보낼 것
    • PUT: 클라이언트에서 서버로 보낸 데이터를 지정한 이름의 리소스로 저장할 것
    • DELETE: 지정한 리소스를 서버에서 삭제할 것
    • POST: 클라이언트 데이터를 서버 게이트웨이 애플리케이션으로 보낼 것
    • HEAD: 지정한 리소스에 대한 응답에서, HTTP 헤더 부분만 보낼 것

    상태 코드

    HTTP 응답 메시지는 상태 코드와 함께 반환된다. 상태 코드는 클라이언트에게 요청이 성공했는지 혹은 추가 조치가 필요한지 알려주는 세 자리 숫자다.

    • 2xx: 좋다.
    • 3xx: 리다이렉션이 이루어져야 함
    • 4xx: 요청이 올바르지 않음

    메시지

    HTTP 메시지는 다음의 세 부분으로 이루어진다.

    // 요청 메시지
GET /test/hello.txt HTTP/1.0   시작줄
-----------------------------------
Accept: text/*                  헤더
Accept-Language: en, fr

// 응답 메시지
HTTP/1.0 200 OK                시작줄
-----------------------------------
Content-type: text/plain        헤더
Content-length: 21
-----------------------------------
Hi! I'm a message!             메시지
    • 시작줄: 무엇을 해야 하는지 / 무슨 일이 일어났는지
    • 헤더: 0개 이상의 헤더 필드가 이어짐. 각 헤더 필드는 쌍점(:)으로 구분되어 이름과 값으로 구성됨. 헤더는 빈 줄로 끝난다.
    • 본문: 빈 줄 다음에는 어떤 종류의 데이터든 들어갈 수 있는 메시지 본문이 필요에 따라 올 수 있다.

    TCP 커넥션

    TCP/IP

    HTTP는 애플리케이션 계층 프로토콜이다. 네트워크 통신의 핵심적인 세부사항에 대해서 신경 쓰지 않는다. 대신 대중적이고 신뢰성 있는 인터넷 전송 프로토콜인 TCP/IP에게 맡긴다. TCP는 다음을 제공한다.

    • 오류 없는 데이터 전송
    • 순서에 맞는 전달
    • 조각나지 않는 데이터 스트림 인터넷은 전 세계의 컴픁와 네트워크 장치들 사이에서 대중적으로 사용되는 TCP/IP에 기초하고 있다. TCP/IP는 패킷 교환 네트워크 프로토콜의 집합이다. 각 네트워크와 하드웨어의 특성을 숨기고, 어떤 종류의 컴퓨터나 네트워크든 서로 신뢰성 있는 의사소통을 하게 해 준다. 일단 커넥션이 맺어지면, 클라이언트와 서버 컴퓨터 간에 교환되는 메시지가 없어지거나, 손상되거나, 순서가 뒤바뀌어 수신되는 일은 없다.

    접속, IP 주소 그리고 포트번호

    클라이언트가 서버에 메시지를 전송할 수 있게 되기 전에, IP(Internet Protocol) 주소와 포트번호를 사용해 TCP/IP 커넥션을 맺어야 한다. IP 주소와 포트번호는 URL을 통해 알 수 있다.

    // case 1
http://207.200.81.31:80/index.html
// case 2
http://www.naver.com:80/index.html
// case 3
http://www.naver.com/index.html

    첫 번재 케이스는 IP 주소(207.200.81.31)와 포트번호(80)가 그대로 드러나있다. 두 번째 케이스는 숫자로 된 IP 주소가 없고 문자로 된 도메인(혹은 호스트 명, www.naver.com)을 갖고 있다. 호스트 명은 DNS(Domain Name Service)를 통해 IP로 변환 될 수 있다. 세 번째 케이스는 포트번호가 없다. 이 경우엔 HTTP 기본 포트인 80이 생략된 경우로 보면 된다.

    IP 주소와 포트번호를 알았으면 클라이언트는 TCP/IP로 통신할 수 있다. 순서는 다음과 같다.

    1. 브라우저는 서버의 URL에서 호스트 명을 추출한다.
    2. 브라우저는 서버의 호스트 명을 IP로 변환한다.
    3. 브라우저는 URL에서 포트번호를 추출한다.
    4. 브라우저는 서버와 TCP 커넥션을 맺는다.
    5. 브라우저는 서버에 HTTP 요청을 보낸다.
    6. 서버는 브라우저에 응답을 돌려준다.
    7. 커넥션이 닫히면, 브라우저는 문서를 보여준다.

    프로토콜 버전

    HTTP 프로토콜은 버전이 여러가지다.

    • HTTP/0.9
      결함이 다수 존재. GET 메서드만을 지원. 멀티미디어 콘텐츠에 대한 MIME 타입이나, HTTP 헤더, 버전 번호를 지원하지 않음. 간단한 HTML 객체를 받아오기 위해 만들어졌다.
    • HTTP/1.0
      처음으로 널리 쓰이기 시작한 버전. HTTP 헤더, 추가 메서드, 멀티미디어 객체 처리를 추가했다. 시각적으로 매력적인 웹페이지와 상호작용하는 폼을 실현. 그러나 잘 정의된 명세는 아니다.
    • HTTP/1.1
      HTTP 설계의 구조적 결함 교정, 두드러진 성능 최적화, 잘못된 기능 제거에 집중했다. 더 복잡해진 웹 애플리케이션과 배포를 지원한다.
    • HTTP/2.0
      HTTP/1.1 성능 문제를 개선하기 위해 구글의 SPDY 프로토콜을 기반으로 설계 됨.

    웹의 구성요소

    • 프락시
      클라이언트와 서버 사이에 위치한 HTTP 중재자
    • 캐시
      많이 찾는 웹페이지를 클라이언트 가까이에 보관하는 HTTP 창고
    • 게이트웨이
      다른 애플리케이션과 연결된 특별한 웹 서버
    • 터널
      단순히 HTTP 통신을 전달하기만 하는 특별한 프락시
    • 에이전트
      자동화된 HTTP 요청을 만드는 준지능적 웹클라이언트

    프락시

    프락시는 클라이언트와 서버 사이에 위치하여, 클라이언트의 모든 요청을 받아 서버에 전달한다. 이 때, 보통은 요청을 수정해서 전달한다. 프락시는 주로 보안을 위해 사용된다. 모든 웹 트래픽 속에서 신뢰할 만한 중재자 역할을 하는 것이다. 또한 요청과 응답을 필터링한다.

    캐시

    웹 캐시와 캐시 프락시는 자신을 거쳐 가는 문서들 중 자주 찾는 것의 사본을 저장해 두는, 특별한 종류의 HTTP 프락시 서버이다. 다음번에 클라이언트가 같은 문서를 요청하면 그 캐시가 갖고 있는 사본을 받을 수 있다.

    클라이언트는 멀리 떨어진 서버보다 근처의 캐시에서 훨씬 더 빨리 문서를 다운 받을 수 있다.

    게이트웨이

    게이트웨이는 다른 서버들의 중재자로 동작하는 특별한 서버다. 게이트웨이는 주로 HTTP 트래픽을 다른 프로토콜로 변환하기 위해 사용된다. 언제나 스스로가 리소스를 갖고 있는 진짜 서버인 것처럼 요청을 다룬다. 클라이언트는 자신이 게이트웨이와 통신하고 있음을 알아채지 못할 것이다.

    HTTP/FTP 게이트웨이는 FTP URI에 대한 HTTP 요청을 받아들인 뒤, FTP 프로토콜을 이용해 문서를 가져온다.

    HTTP Client  ← HTTP →  HTTP/FTP Gateway  ← FTP →  FTP Server

    터널

    터널은 두 커넥션 사이에서 로우(raw) 데이터를 열어보지 않고 그대로 전달해주는 HTTP 애플리케이션이다. HTTP 터널은 주로 비 HTTP 데이터를 하나 이상의 HTTP 연결을 통해 그대로 전송해주기 위해 사용된다.

    에이전트

    에이전트는 사용자를 위해 HTTP 요청을 만들어주는 클라이언트 프로그램이다. 대표적으로 웹 브라우저가 있다.
